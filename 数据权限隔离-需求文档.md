# DataAgent 数据权限隔离 — 需求文档

> 版本: v1.0  
> 日期: 2026-06-19  
> 状态: 待实现

---

## 1. 概述

DataAgent 作为多租户 CDP 平台，需要对所有对象实施三层数据权限隔离：租户级、部门级、用户级。默认「谁创建谁可见」，支持创建者/主管主动分享给部门或全租户。

---

## 2. 三层隔离

### 2.1 租户隔离（L1）

| 项目 | 说明 |
|------|------|
| 规则 | 租户 A 不能查看、操作租户 B 的任何数据 |
| 实现方式 | 所有业务表 `tenant_id` 字段；API 层从登录 Token 提取 tenant_id，不再接受客户端传入 |
| 当前状态 | ✅ DB 层已有 `tenant_id`；❌ API 层未校验 Token 与请求 tenant_id 一致性 |

### 2.2 部门隔离（L2）

| 项目 | 说明 |
|------|------|
| 规则 | 同一租户内，不同部门的资源默认不可见；可主动分享给指定部门 |
| 实现方式 | 资源表增加可见范围字段；查询时按部门过滤 |
| 数据模型 | 复用现有 `teams` + `team_members` |

### 2.3 用户隔离（L3）

| 项目 | 说明 |
|------|------|
| 规则 | 同一部门内，用户创建的资源默认只有自己可见 |
| 实现方式 | 资源表 `created_by` 字段 → 查询时追加 `created_by = current_user_id` 过滤 |
| 例外 | 被分享后不受此限制 |

---

## 3. 分享模型

```
┌─────────────────────────────────────┐
│         资源创建（默认）            │
│   可见范围: 仅创建者 (user 级)      │
└─────────────┬───────────────────────┘
              │ 主动分享
     ┌────────┴────────┐
     ↓                 ↓
┌─────────┐      ┌──────────┐
│ 部门可见 │      │ 全租户可见│
│ (team)  │      │ (tenant) │
└─────────┘      └──────────┘
```

### 3.1 分享粒度

| 粒度 | 说明 | 示例 |
|------|------|------|
| **资源级** | 每个资源单独设置可见范围 | Segment A 分享给华东，Segment B 仅自己 |
| **策略级** | 部门/角色默认策略，该部门创建的资源自动应用 | 「华东团队创建的 Segment 默认全员可见」 |

两种粒度同时支持，资源级优先于策略级。

### 3.2 分享权限

| 角色 | 可执行的分享操作 |
|------|------------------|
| 资源创建者 | 分享自己的资源给部门或全租户 |
| 部门主管 | 分享部门内任意资源 |
| 租户管理员 | 分享租户内任意资源 |

## 5. 数据模型设计

### 5.1 资源可见范围字段

在所有受控资源表上增加字段（或通过独立关联表）：

```sql
-- 方案 A：资源表直接加字段（简单，推荐）
ALTER TABLE user_groups ADD COLUMN visibility ENUM('user','team','tenant') DEFAULT 'user';
ALTER TABLE user_groups ADD COLUMN created_by BIGINT NOT NULL;
-- team_ids 存 JSON 数组，空=未分享给部门
ALTER TABLE user_groups ADD COLUMN shared_team_ids JSON DEFAULT '[]';

-- 方案 B：独立可见范围表（解耦，灵活）
CREATE TABLE resource_visibility (
    resource_type VARCHAR(64) NOT NULL,  -- 'segment', 'group', 'tag', ...
    resource_id   VARCHAR(64) NOT NULL,
    tenant_id     BIGINT NOT NULL,
    visibility    ENUM('user','team','tenant') DEFAULT 'user',
    created_by    BIGINT NOT NULL,
    shared_team_ids JSON DEFAULT '[]',
    PRIMARY KEY (resource_type, resource_id)
);
```

### 5.2 策略级默认表

```sql
CREATE TABLE visibility_policies (
    tenant_id    BIGINT NOT NULL,
    team_id      BIGINT DEFAULT NULL,       -- NULL = 全租户默认
    resource_type VARCHAR(64) DEFAULT '*',  -- '*' = 所有类型
    default_visibility ENUM('user','team','tenant') DEFAULT 'user',
    created_at   DATETIME DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_policy (tenant_id, team_id, resource_type)
);
```

## 6. 查询执行流程

```
用户请求 → Token 解析 → 确认 tenant_id + user_id + teams
         ↓
    权限判断:
    ① 资源 visibility = 'tenant' → 全租户可见，直接查
    ② 资源 visibility = 'team' → 用户所在 teams ∩ 资源 shared_team_ids ≠ ∅ → 可见
    ③ 资源 visibility = 'user' → created_by = current_user_id → 可见
    ③ → 否则 403
```

### 6.1 SQL 层面

```python
def build_visibility_filter(current_user, resource_table):
    """返回追加的 WHERE 条件 + 参数"""
    user_id = current_user["id"]
    team_ids = current_user["team_ids"]  # [9001, 9002]
    
    # 可见条件: 我是创建者 OR 分享了给我所在的团队 OR 全租户可见
    filter_sql = """
        AND (
            r.visibility = 'tenant'
            OR r.created_by = %(current_user_id)s
            OR (r.visibility = 'team' AND JSON_OVERLAPS(r.shared_team_ids, %(current_team_ids)s))
        )
    """
    params = {
        "current_user_id": user_id,
        "current_team_ids": json.dumps(team_ids),
    }
    return filter_sql, params
```

## 9. 注意事项

- 现有的种子数据（user_groups, tags 等）迁移时，`created_by` 默认设为 admin (90001)，`visibility` 默认设为 `tenant`
- 历史数据兼容：迁移前存在的资源，统一设为 `tenant` 可见，避免现有功能中断
- 部门隔离依赖 `team_members` 表准确维护用户-部门关系
